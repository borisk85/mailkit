# MailKit — технические меры против злоупотреблений

Документ описывает технический слой защиты от сценария "один клиент
нарушил политику Brevo и весь наш общий аккаунт заморозили". Содержит
три обязательных для MVP механизма плюс план действий при инциденте
плюс подготовку запасного SMTP-провайдера на случай полной заморозки.

Связанные документы: [docs/SECURITY.md](SECURITY.md) — общая модель
разделяемого аккаунта Brevo; [docs/LEGAL_PROTECTIONS.md](LEGAL_PROTECTIONS.md)
— юридический слой защиты; [docs/GUARANTEE_POLICY.md](GUARANTEE_POLICY.md)
— политика возврата денег.

---

## 1. Угроза

Модель разделяемого корпоративного аккаунта Brevo для всех клиентов —
операционное решение для MVP чтобы избежать индивидуальной настройки
на каждого клиента. Это даёт нам скорость и дешевизну, но создаёт
единую точку отказа.

Основные сценарии угрозы:

1. **Один клиент отправил спам или фишинг.** Получатели жалуются.
   Brevo детектит, замораживает наш аккаунт целиком. Все клиенты
   теряют возможность отправлять почту через SMTP-ретрансляцию.
   Восстановление — переписка с поддержкой Brevo, срок от нескольких
   дней до нескольких недель.

2. **Один клиент превысил лимит плана Brevo** (например 300 писем в
   сутки на бесплатном плане). Brevo начинает отклонять отправку,
   страдают все клиенты на аккаунте.

3. **Высокий процент отказов (bounces) с одного домена.** Brevo
   автоматически помечает аккаунт как низкокачественный, понижает
   репутацию IP-адресов отправки, все клиенты получают ухудшение
   доставляемости.

4. **Жалобы на спам (complaints) от получателей.** Получатель нажимает
   "пометить как спам" в своей почте. Метрика комплейнтов растёт,
   аккаунт попадает под санкции.

Первые два сценария предотвращаются лимитами на нашей стороне. Третий
и четвёртый — мониторингом метрик через API Brevo и автоматической
приостановкой проблемного домена до того как Brevo заметит.

## 2. Слой 1 — Лимиты отправки на домен клиента

### 2.1 Пороги

Для MVP устанавливаются следующие жёсткие лимиты:

- **500 писем в сутки** на один домен клиента. Достаточно для
  транзакционных писем, подтверждений, личной переписки. Существенно
  ниже порогов при которых Brevo начинает задавать вопросы.
- **50 писем в час** на один домен. Защита от внезапных всплесков
  отправки, например если клиент написал скрипт для массовой рассылки.
- **5 писем в минуту** на один домен. Защита от автоматической
  массовой отправки.

Для клиентов купивших подписку мониторинга (3 доллара в месяц) лимиты
могут быть увеличены в индивидуальном порядке — это делается через
запись в отдельном поле таблицы purchases (например `custom_limits`
jsonb с overrides).

### 2.2 Измерение

Источник данных — API статистики Brevo endpoint
`GET /v3/smtp/statistics/events` с фильтром по sender-домену и временному
окну. Этот API возвращает список событий (requested, delivered,
opened, bounced, complained) с временными метками.

Для счётчика отправленных писем суммируем события `requests` за
нужное временное окно.

Кэширование: делать API-вызов на каждое письмо нецелесообразно (лишняя
нагрузка + задержки). Вместо этого — локальный счётчик в таблице
`send_counters` с колонками `domain, window_type, window_start,
count`. Периодически синхронизировать с Brevo (каждые 5-10 минут
фоновый процесс) чтобы избежать расхождений.

### 2.3 Проверка перед отправкой

Все исходящие письма через наш SMTP проходят через нашу же прокси-логику
перед достижением Brevo. Это уже так для транзакционных писем которые
мы отправляем от имени MailKit (welcome, refund notification, etc).
Для писем клиентов через Gmail Send-As — клиенты отправляют напрямую
через Brevo SMTP-relay, мы физически не видим отдельные письма.

Тут компромисс архитектуры: мы не можем блокировать письма клиента в
момент отправки, потому что они идут мимо нашего кода. Но мы можем:

1. **Мониторить счётчик postfactum** (каждые 5 минут через API
   статистики) и при превышении лимита — приостанавливать домен в
   Brevo через API (endpoint `POST /v3/senders/domains/{name}/deactivate`
   или аналогичный — нужно уточнить в документации).
2. **Рассылать уведомление клиенту** "вы превысили лимит отправки,
   свяжитесь с поддержкой" через отдельный транспорт (например наш
   административный email).
3. **Логировать инцидент** в таблицу `abuse_events` для аудита и
   триггера юридических действий если это регулярно.

### 2.4 Текст клиенту при блокировке

Шаблон письма отправляется клиенту при срабатывании лимита:

```
Тема: Отправка почты с <домен> временно приостановлена

Привет,

Мы временно приостановили отправку писем с вашего домена <домен>
потому что за последние <период> было отправлено <N> писем. Это
превышает наш лимит в <лимит> писем за этот период.

Лимит защищает всех пользователей MailKit — без него один
недобросовестный отправитель может навредить доставляемости всем
клиентам на нашей инфраструктуре.

Если вам нужен увеличенный лимит для легитимной отправки (например,
транзакционные письма крупного магазина или массовая рассылка по
подписке) — напишите на support@getmailkit.com с объяснением
сценария. Мы обычно повышаем лимит в течение 24 часов для
обоснованных кейсов.

Отправка будет автоматически возобновлена через <интервал> когда
счётчик сбросится.

— MailKit
```

## 3. Слой 2 — Мониторинг метрик доставляемости

### 3.1 Пороги приостановки домена

Пороги срабатывают на скользящем 7-дневном окне, чтобы сгладить
единичные всплески:

- **Процент отказов (bounce rate) > 5%** — возможная проблема со
  списком получателей или содержимым. Автоматически приостанавливаем
  домен в нашей системе (флаг `suspended` в таблице
  `setup_runs` или `purchases`), письмо клиенту "проблема с
  доставляемостью, свяжитесь с поддержкой".
- **Процент жалоб на спам (complaint rate) > 0.1%** — критично,
  немедленная приостановка. Этот порог установлен потому что Brevo
  сама начинает вмешиваться при 0.3-0.5%.
- **Процент отписок (unsubscribe rate) > 2%** — сигнал о
  неожидаемом содержимом. Уведомление клиенту (не блокировка) —
  "похоже ваши получатели не ожидают ваши письма, проверьте список и
  согласие".

### 3.2 Измерение

Источник — тот же API Brevo `GET /v3/smtp/statistics/events` и
`GET /v3/smtp/statistics/aggregatedReport`. Фоновый процесс раз в
час собирает метрики за последние 7 дней для каждого активного
домена, вычисляет проценты, сравнивает с порогами.

Таблица `deliverability_snapshots`:

```
id uuid primary key
domain text not null
measured_at timestamptz not null
requests_count integer
bounced_count integer
complained_count integer
unsubscribed_count integer
bounce_rate numeric(5,2)
complaint_rate numeric(5,2)
unsubscribe_rate numeric(5,2)
action_taken text  -- null | 'warned' | 'suspended'
```

Хранится для аудита и для трендов. Очищается после 90 дней.

### 3.3 Действия при срабатывании

При превышении порога жалоб:
1. Устанавливаем флаг `suspended_at` на соответствующей записи в
   таблице `purchases` или `setup_runs`
2. Вызываем Brevo API для деактивации домена-отправителя (чтобы
   отправка физически прекратилась)
3. Отправляем клиенту уведомление (шаблон ниже)
4. Создаём запись в `abuse_events` для аудита
5. При повторном срабатывании в течение 30 дней на том же домене —
   постоянная блокировка, требует ручного разбирательства поддержки

Шаблон уведомления клиенту при срабатывании жалоб:

```
Тема: Проблема с доставляемостью на <домен>

Привет,

За последние 7 дней процент жалоб на спам на ваши письма с <домен>
составил <Х%> — это превышает наш порог (0.1%). Мы временно
приостановили отправку с этого домена чтобы защитить репутацию
нашей SMTP-инфраструктуры и всех других клиентов MailKit.

Типичные причины высокого процента жалоб:
- Отправка на старый или купленный список контактов
- Отсутствие согласия получателей на рассылку
- Неочевидная отвязка (отсутствие ссылки "отписаться")
- Содержимое не соответствующее заявленной теме

Рекомендации:
1. Используйте только согласованные контакты (double opt-in)
2. Добавляйте ссылку на отписку в каждое письмо
3. Прогревайте репутацию домена постепенно (не более 50 писем в день
   в первую неделю после настройки)

Если вы считаете блокировку ошибочной — напишите на
support@getmailkit.com, разберёмся индивидуально.

— MailKit
```

## 4. Слой 3 — План действий при инциденте

Если несмотря на все защитные меры Brevo заморозит наш аккаунт —
готовый сценарий действий.

### 4.1 Обнаружение

Сигналы что аккаунт заморожен:
- API Brevo возвращает ошибки 403 на все запросы с нашим ключом
- SMTP-ретрансляция возвращает ошибки аутентификации 535 на уровне Gmail
  (клиенты видят "Sent mail could not be delivered")
- Письмо от Brevo в наш поддерживающий inbox о нарушении

Настраиваем мониторинг:
- Автопроверка API каждые 5 минут с алертом на наш email при
  3+ подряд 403 ответах
- Подписка на input email box для notification от Brevo с автопереадресацией
  на владельца

### 4.2 Первый час — стабилизация

1. Включаем флаг `global_smtp_paused` в переменных окружения или
   отдельной таблице настроек. Наш Gmail Send-As wizard при этом
   показывает клиентам новым "настройка временно недоступна, работаем
   над инцидентом".
2. Рассылаем всем активным клиентам уведомление через отдельный канал
   (не Brevo SMTP!) о временном простое. Для этого держим секретный
   webhook-URL на сторонний сервис (например Resend или Amazon SES
   sandbox — бесплатно до 200 писем в день, хватит на уведомление).
3. Пост в статус-странице (если есть) или в социалках.

### 4.3 Первые сутки — диагностика и коммуникация с Brevo

1. Открываем тикет в поддержку Brevo с максимально полным контекстом:
   идентификатор аккаунта, домены клиентов, описание сервиса, просьба
   идентифицировать проблемного отправителя.
2. Во время ожидания ответа — анализируем наши логи `abuse_events` и
   `deliverability_snapshots` чтобы самим понять кто мог вызвать
   проблему.
3. При идентификации — удаляем виновный домен из нашей системы,
   предоставляем доказательства Brevo.
4. Рассылаем обновление клиентам о статусе.

### 4.4 Первая неделя — запасной план

Если восстановление занимает больше 48 часов — переключаем активных
клиентов на запасного SMTP-провайдера (см. раздел 5). Это требует:

1. Обновить SPF-записи в DNS всех клиентских доменов — заменить
   `include:spf.brevo.com` на `include:<новый провайдер>`. Наш
   код через Cloudflare API может сделать это массово.
2. Обновить переменные окружения Vercel на новый SMTP-хост и
   реквизиты.
3. Пересоздать Gmail Send-As на новом SMTP-relay — для этого нужно
   прислать клиентам новые реквизиты и попросить обновить в Gmail.
   Это ручной шаг для клиента, 3-5 минут работы.

Реалистичный срок миграции на запасной — 24-48 часов с момента
решения переключаться.

## 5. Слой 4 — Подготовка запасного провайдера (после запуска)

Не входит в MVP, но должен быть подготовлен в течение первого месяца
после запуска.

### 5.1 Выбор провайдера

Кандидаты:
- **Amazon SES** — бесплатно до 62 000 писем в месяц при отправке с
  EC2, или 0.10 доллара за 1000 писем. Промышленный стандарт,
  надёжность высокая. Требует аккаунт AWS и верификацию отправляющего
  домена (аналогично Brevo).
- **Mailgun** — фримиум-план 5000 писем в месяц бесплатно, далее
  платные тарифы. Хорошая доставляемость. Удобный API.
- **Postmark** — специализируется на транзакционной почте, отличная
  доставляемость. Платно с первого письма (10 долларов в месяц за
  10 000 писем).

Рекомендация — **Amazon SES как основной резерв** по стоимости и
надёжности, **Mailgun как вторичный резерв** на случай проблем с AWS.

### 5.2 Подготовка

Должно быть сделано в течение 30 дней после запуска:
1. Регистрация аккаунта AWS, прохождение SES-верификации (может
   занять несколько дней, AWS проверяет use case)
2. Тестовая отправка через SES с тестового домена, проверка работы
3. Написание кода переключения провайдера (переменная окружения
   `SMTP_PROVIDER=brevo|ses|mailgun` плюс соответствующие адаптеры в
   коде)
4. Тестирование массовой смены SPF-записей через Cloudflare API (на
   тестовом домене)
5. Документирование процедуры миграции в отдельном runbook'е

Стоимость подготовки — время разработчика примерно 8-12 часов,
денежные расходы на AWS минимальные (возможно 5-10 долларов за
тестирование).

## 6. Что входит в MVP (до запуска)

Обязательные задачи для запуска:

1. **Тикет — Лимиты отправки на домен.** Таблица `send_counters`,
   фоновый синхронизатор с Brevo, деактивация домена при превышении,
   уведомление клиенту. Оценка 1-2 дня разработки.
2. **Тикет — Мониторинг доставляемости.** Таблица
   `deliverability_snapshots`, часовой сборщик метрик, пороги
   приостановки, уведомления клиенту. Оценка 1-2 дня разработки.
3. **Тикет — Страница условий пользования /terms.** Контент из
   [docs/LEGAL_PROTECTIONS.md](LEGAL_PROTECTIONS.md) раздел 2.1 (EN)
   и 2.2 (RU), статическая страница Next.js. Оценка 2-3 часа
   разработки.
4. **Согласие клиента на условия** — настройка в Lemon Squeezy
   (обязательная галочка или ссылка в описании товара), задача
   владельца не разработчика.

## 7. Что в бэклог (после запуска)

Задачи второй очереди:

1. **Amazon SES аккаунт и верификация** — 1-2 недели после запуска
   (задача владельца по регистрации + разработчик на тестирование)
2. **Адаптеры SMTP-провайдеров в коде** — 2-4 недели после запуска
   (разработчик)
3. **Runbook миграции SPF и SMTP** — отдельный документ после
   реализации адаптеров
4. **Оформление юридического лица** — при 100+ клиентах или первом
   инциденте (задача владельца)
5. **E&O insurance** — при наличии юрлица (задача владельца)
6. **Автоматический webhook статуса в социальные сети** — при
   накоплении клиентской базы

## 8. Приоритеты реализации пре-запуск

Порядок работы разработчика до запуска:

1. Завершение текущего тикета #7 (приём платежей через Lemon Squeezy)
2. Мини-тикет — страница /terms с контентом из LEGAL_PROTECTIONS.md
3. Тикет — лимиты отправки на домен
4. Тикет — мониторинг доставляемости
5. Финализация лендинга (этап 3 тикета #11)
6. Запуск

Общая оценка оставшейся работы до запуска с учётом новых задач:
6-8 дней сконцентрированной разработки.
