Comparison context
==================
Post-#4b landing baseline (prod, warm, n=5): EN 75 / RU 85.
/app/setup has no stored prod baseline — mock=* bypass is preview-only
and authenticated /app/setup cannot be exercised by headless Lighthouse
against prod without the OAuth session.

This measurement is therefore smoke-level, not a regression comparator.

Per-locale run-over-run spread: EN 64-78 (Δ14), RU 64-78 (Δ14) — both
wider than our anti-regression 5-point threshold, confirming this is
preview-lambda variance at the same level previously documented for
landing in docs/investigation-2026-04-22/FINDINGS.md.

Absolute thresholds (CLAUDE.md Performance quality gate):
  Performance ≥90  — not met (medians 65/69; typical for JS-heavy
                     authenticated pages + mock rehydration path)
  LCP <2.5s        — not met (2.73s / 2.77s)
  TBT <200ms       — not met (306 / 247 ms)
  FCP <1.8s        — met (1.15s / 1.19s)
  CLS <0.1         — met (0.000 / 0.000)

Reasonable given:
- /app/setup mounts the full wizard client component (many icons, multiple
  state branches, useTransition)
- preview lambdas + simulate throttling inflate TBT run-over-run
- mock bypass path renders the wizard from RSC with full state seeded,
  which is unlike real user flow (where state arrives incrementally)

Decision
========
NOT merging on LH alone. Per docs/LESSONS_LEARNED.md "Preview vs prod
Lighthouse: разные измерительные среды", preview LH is non-gating with
cold-start variance up to ×15 run-over-run. The 14-point spread we see
above is exactly that pattern.

No dedicated perf fix PR for etap 2. Systemic perf work stays in the
backlog as attempt-3 (next-intl pick, selective dynamic route segments,
bundle analyzer) — covers /app/setup along with landing.

If post-merge prod /app/setup instrumentation (auth'd flow, e.g. once we
ship analytics) reveals the median below usable thresholds in real-user
data, re-open as a dedicated investigation.
